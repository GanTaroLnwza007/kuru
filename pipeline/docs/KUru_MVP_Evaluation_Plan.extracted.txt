1. Purpose
This document defines the evaluation plan for KUru's MVP recommendation system. It specifies which use cases are in scope for the current development phase, which have been explicitly cut and why, the quantitative and qualitative metrics used to assess each in-scope feature, the methodology for constructing the evaluation test set, and the phased timeline for conducting evaluations.
This plan is the single reference for what constitutes a passing MVP. Features not listed as in scope should not be evaluated or built in the current phase.

The table below summarises all use cases from the SRS, their inclusion status in the MVP, and the evaluation method applied. Green rows are fully in scope. Yellow rows are partially in scope with explicit capability boundaries. Red rows are cut from the MVP and deferred to Phase 2.

4. Evaluation Metrics
KUru's MVP is evaluated across two categories: automated quantitative metrics applied to the AI pipelines, and human-centred qualitative metrics applied via user testing. All targets are taken directly from the SRS Section 4.3.3 unless otherwise noted.
4.1 MRR and NDCG@5 — Recommendation Ranking Quality
Mean Reciprocal Rank (MRR) measures where the first correct program appears in the ranked output. For a profile whose correct program appears at rank 1, the reciprocal rank is 1.0; at rank 2 it is 0.5; at rank 5 it is 0.2. MRR is the mean of these values across all test profiles. A target of MRR > 0.60 means the correct program appears within the top 2 results on average.
NDCG@5 (Normalised Discounted Cumulative Gain at rank 5) penalises correct programs that appear lower in the list more heavily than those at the top. It is reported alongside MRR to capture cases where MRR is high but lower-ranked results are poor.
Both metrics are computed on the manually curated test set described in Section 5. Because Pipeline B2 is the only signal in the simplified MVP (Neo4j Pipeline B1 is not yet populated), the baseline comparison is a keyword-only ilike search, not the full two-pipeline system described in the SRS.
4.2 RAGAS — RAG Pipeline Quality
The RAGAS framework provides automated evaluation of the retrieval-augmented generation pipeline without requiring human annotation of every answer. Two metrics are used:
Faithfulness measures whether every claim in a generated answer or explanation can be traced back to a retrieved มคอ.2 chunk. A faithfulness score of 1.0 means every claim is grounded; 0.0 means all claims are hallucinated. Target: > 0.80.
Answer Relevancy measures the semantic similarity between the generated answer and the original question, penalising off-topic or incomplete responses. Target: > 0.75.
RAGAS is applied to UC-04 (chatbot answers) and UC-08 (recommendation explanations) independently. Both use the same pgvector corpus and Gemini 2.5 Flash generation model, so a shared faithfulness baseline is expected.
4.3 SUS Score and Task Completion Rate
The System Usability Scale (SUS) is a 10-item Likert questionnaire administered after each user testing session. Scores range from 0 to 100. A score above 70 is considered acceptable; above 85 is excellent. The target of SUS > 70 applies across all MVP features collectively.
Task completion rate is measured specifically for UC-01: the percentage of test participants who complete all 12 steps of the RIASEC elicitation without abandoning the flow. A target of > 80% completion indicates the interface is accessible to the high school student audience.
5. Evaluation Test Set
Automated metrics (MRR, NDCG@5, RAGAS) require a curated test set. The test set is constructed manually by the development team before the evaluation sprint and validated by the project advisor or senior SKE students.
5.1 Profile Construction Guidelines
Each synthetic profile must represent a distinct, realistic RIASEC type combination. The following distribution is recommended to cover the space of KU faculties:
High R + I: Engineering faculties (CPE, ME, EE)
High I + A: Architecture, Design
High S + E: Business, Education
High C + E: Accounting, Finance
High I + C: Science, Statistics
Mixed profiles (no dominant type): used to test robustness of the ranking under ambiguity
Profiles with a dealbreaker filter applied (one dimension zeroed out) must be included to verify that the filter correctly suppresses programs dependent on that dimension.
5.2 Ground Truth Validation
Ground-truth program matches are validated by at least one of the following:
The project advisor (Aj. Jitti Niramitranon), who has domain knowledge of KU program content
Senior SKE students (Year 3 or Year 4) who have direct experience of KU program selection
Cross-reference with the program's PLO keywords in the ingested มคอ.2 corpus
A profile is included in the final test set only when the ground-truth match has been confirmed by at least one validator. Disputed profiles are excluded.
6. Evaluation Timeline
Evaluations are conducted in five phases aligned with the build order defined in the project schedule. Each phase produces a concrete output that feeds into the next.
Phase 5 (User Testing) is the final gate before project submission. All automated metric targets must be met before user testing sessions are scheduled, so that participants interact with a system that has already passed its quantitative thresholds.

UC-10 Program Comparison View — side-by-side comparison of pinned programs
UC-11 Curriculum Timeline Visualiser — year-by-year course structure display
UC-12 Portfolio Readiness Checker — document upload and gap analysis
Neo4j Pipeline B1 — PLO graph-based curriculum matching (schema exists, population deferred)
PLO spider chart student profile overlay — static PLO chart only, no RIASEC vector overlay
Standalone TCAS guide page — TCAS data accessible via chatbot and program detail page only
Saved Profile Dashboard — requires persistent login and session management
TCAS Deadline Tracker — requires persistent login and push notification infrastructure
Collaborative filtering across users — deferred pending sufficient interaction log data
