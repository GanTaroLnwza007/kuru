  PART A — AI-Enabled System Design  


1. Project Definition
Task A1.1 — Project Title and Motivation
Project Title
KUru: KU Curriculum & PLO Navigator


Problem Domain & Target Users
Kasetsart University (KU) prospective students — high school leavers preparing for TCAS applications — need to query specific information about KU programs: credit requirements, course structure, Program Learning Outcomes (PLOs), and admission criteria. This information exists in official มคอ.2 curriculum documents: 20–100 page formal Thai-language PDFs, one per program. Students currently must read them manually. There is no searchable interface.


Why AI/ML is Necessary (vs. Rule-Based Approach)
* Vocabulary mismatch — students ask in colloquial Thai or English; PDFs use formal academic Thai terminology. Exact keyword matching cannot bridge this gap (e.g., "เรียนอะไรบ้าง" vs. "โครงสร้างหลักสูตร").
* Cross-lingual retrieval — the student population queries in both Thai and English, but all curriculum documents are in Thai. A rule-based system has no mechanism for cross-language matching.
* Synthesis across sections — questions like "What are the minimum credits for general education in Computer Engineering?" require combining a credit table, a section header, and program identity from different parts of the document. Keyword search returns fragments; an LLM synthesises them into a coherent answer.


Expected Impact & Value
Students can resolve curriculum and admission questions in under 5 seconds through natural language, without downloading or reading PDFs. This reduces the information barrier to informed TCAS program selection across the cleaned Bangkhen corpus of 57 KU program records.

Task A2.2 — AI Component Design
A2.2.1 — AI Problem Identification
The system uses AI to solve two specific sub-problems:
* Semantic retrieval across a multilingual document corpus — given a Thai or English question, find the most relevant passages from มคอ.2 PDFs across the cleaned 57-program corpus. This cannot be solved with keyword search alone due to vocabulary and language gaps.
* Answer synthesis and grounding — given retrieved passages, generate a fluent, accurate answer that cites its sources and does not hallucinate beyond the provided context. This requires an LLM with a strict citation-grounded prompt.

Feedback → Retraining Pipeline
For a RAG pipeline, re-ingestion is the functional equivalent of model retraining — updating the indexed vector store with better-extracted chunks directly improves retrieval quality without changing model weights. The pipeline is triggered manually (not automated CI/CD):
1. User thumbs-down accumulates in Supabase feedback table.
2. Daily aggregation: programs where negative_rate > 30% are queued for re-ingestion.
3. If chunk_count < 100: re-ingest full มคอ.2 PDF with OCR_MODEL=gemini-2.5-flash (higher-quality OCR).
4. If chunk_count ≥ 100: investigate chunking quality (tabular extraction, section boundary detection).
5. Run VACUUM ANALYZE on chunks table → rebuild IVFFlat index.
6. Re-run eval set for affected programs → compare before/after score → log in eval_results.md.

Task B-2 — Model Training Implementation
Data Preprocessing (all steps in order)
7. Load eval CSV (data/eval_set_v2.csv, 50 rows); validate required columns: question, ground_truth_answer, question_type, program_id, section_type.
8. Fill missing values: question_type → 'unknown', section_type → 'unknown', program_id → ''.
9. Encode question_type as integer label: 0=admission, 1=curriculum, 2=general, 3=plo.
10. Sample 20 questions with fixed seed (RANDOM_SEED=42, SAMPLE_SIZE=20) for MLflow runs.
11. PDF text extraction — PyMuPDF page.get_text() for native-text PDFs; Gemini Files API OCR fallback for scanned pages (detected by char_count < 100 per page).
12. Character-based chunking — 2,000-char chunks, 200-char overlap, section-boundary aware.
13. Section type classification — deterministic regex: TCAS/GPAX keywords → admission; PLO headers → plo; course code pattern → course; fallback → general.
14. Embedding — intfloat/multilingual-e5-base; docs prefixed 'passage:', queries prefixed 'query:'; 768-dim float32, cosine-normalized.
15. Indexing — bulk insert into Supabase pgvector; VACUUM ANALYZE; IVFFlat index with probes=10.

Task B-3 — Model Fairness Analysis
Applicability Statement
This is not demographic fairness — the dataset contains no protected demographic attributes (age, gender, ethnicity, socioeconomic status). All source material is official public KU curriculum and admission information. The relevant fairness axis is coverage disparity: programs or section types with sparse, OCR-noisy, or table-heavy chunks receive worse answers than well-covered programs. The B3 analysis measures whether the RAG system performs equally across section types and programs.

Bias Mitigation Approaches
* Re-ingestion priority queue: programs with <50 chunks flagged in data/coverage_report.md and queued for re-ingestion with higher-quality OCR model (gemini-2.5-flash).
* PLO query boosting: when is_plo_query=True detected, re-ranker applies +5 score bonus to PLO-tagged chunks over general chunks of similar cosine similarity.
* Honest coverage notes: when no PLO data exists for a queried program, system injects a Thai-language coverage note into the prompt rather than allowing hallucination.
* Campus scope guard: under-served campuses (กำแพงแสน, ศรีราชา) documented as out-of-scope with hard no-data guard message.
* Language-neutral retrieval: multilingual-e5-base encodes Thai and English queries in the same embedding space; both name_th and name_en indexed in program resolver.

Domain Interpretation
* Program name is the most powerful retrieval signal. 'Engineering' (0.0821) and 'Computer' (0.0743) in Case 1; 'วิศวกรรมโยธา' (0.1387) and 'ชลประทาน' (0.1204) in Case 3. In all three cases, recognising the program name focused the search on the correct program's documents.

* 'วิศวกรรมโยธา' and 'ชลประทาน' together achieve the highest individual token importance (0.139 and 0.120), reflecting how both compound parts together distinguish Civil Engineering - Irrigation from plain Civil Engineering.
* Confidence is uniformly HIGH across all three cases because the program was successfully resolved. When resolution fails, similarity typically drops below 0.50.
* Targeted lexical reranking was more impactful for the current production model than simply widening the retrieval window (wider_retrieval top_k=8 vs. production top_k=5).
* Admission queries are consistently harder than curriculum queries across all experiments — reflecting the heterogeneous format of TCAS PDFs.

User Feedback
Thumbs up / thumbs down buttons below each response. Single click — no form required. Writes rating (+1 or -1) to the Supabase feedback table via POST /api/v1/chat/feedback.
